# Ошибки и открытые риски

## Открытые риски

### COM4 снова занят перед загрузкой финальной прошивки

16 июля финальная прошивка успешно собрана, однако три последовательные попытки
`arduino-cli upload` не смогли открыть COM4 (`PermissionError(13)`). Порт и плата
видны системе, но дескриптор удерживается другим процессом; попытка перезапустить
только USB-интерфейс через `pnputil` отклонена Windows без прав администратора.
Запись flash ни разу не начиналась, поэтому установленная на плате предыдущая
версия не повреждена. Перед следующей попыткой нужно закрыть программу, занявшую
COM4, либо физически переподключить плату.

Причина найдена: COM4 удерживал отдельный процесс `serial-monitor.exe`, запущенный
Arduino IDE. После остановки только этого процесса загрузка завершилась успешно,
все области прошли проверку hash, плата перезагружена. Риск закрыт.

### Проверка сна GPS на аппаратуре

Нужно проверить на реальном модуле, что `UBX-RXM-PMREQ` действительно переводит
конкретную плату QUESCAN M10Q в backup и что фронт UART TX стабильно пробуждает
приёмник после сна ESP32. В текущем окружении это невозможно проверить без платы.

### Реальный MTU, скорость и степень сжатия BLE

Прошивка запрашивает MTU 247 и формирует уведомления не более 202 байт. Нужно
проверить согласованный MTU на целевом телефоне, скорость stop-and-wait передачи,
частоту повторов и фактический выигрыш PackBits на горных треках. При малом MTU
потребуется согласование меньшего размера блока через `OpenTransfer`.

### COM4 занят при первой попытке прошивки

15 июля 2026 года плата определилась на COM4, но две попытки `esptool` завершились
до записи flash с `PermissionError(13)`: порт удерживается другим процессом.
Прошивка на устройстве не изменена. Перед повтором нужно закрыть serial monitor,
веб-страницу Web Serial или другую программу, открывшую COM4.

После физического освобождения порта повторная загрузка завершилась успешно:
все области проверены по hash, плата аппаратно перезагружена. Риск закрыт.

### Восстановление файла на SD

Arduino-ESP32 не предоставляет переносимую операцию усечения файла. Поэтому при
обнаружении повреждённого хвоста валидная часть копируется во временный файл,
после чего исходный файл заменяется. Нужно проверить поведение при внезапном
отключении питания во время самого восстановления.

## Найденные и исправленные ошибки

### GPS продолжал собирать точки после достижения лимита цикла

Проверка `pointsBeforeSleep` находилась только внутри общего условия перехода в
Deep Sleep. Если сон блокировали USB, 30-секундное кнопочное окно или активное BLE
при выключенном соблюдении графика, прошивка продолжала опрашивать GPS и записывать
точки сверх установленного количества. Добавлено отдельное состояние завершения
цикла: сразу после последней требуемой записи GPS однократно переводится в standby,
UART закрывается и новые точки больше не принимаются. Разрешённые блокировки теперь
откладывают только сон ESP32, но не остановку GPS.

### Старая координата могла повторяться после потери фикса

TinyGPSPlus сохраняет последнюю валидную локацию после потери GPS-фикса. Проверки
только `location.isValid()` было недостаточно: меняющееся время NMEA могло создать
новую запись со старой координатой. Прошивка теперь принимает только координату,
зафиксированную парсером не более 250 мс назад. Строка GGA без фикса остаётся
доступной для диагностики, но в C6T2 не попадает.

### Повреждение общего кэша Arduino после прерванной сборки

Первая контрольная компиляция новой прошивки была остановлена внешним тайм-аутом,
из-за чего объект `Esp.cpp.o` в общем кэше Arduino оказался усечён. Повторная
сборка закономерно завершилась ошибкой `core.a: file truncated`. Для проверки
назначен чистый `--build-path` внутри проекта; полная сборка после этого прошла.
Для последующих контрольных сборок следует использовать отдельный build-path или
удалять только конкретный повреждённый каталог кэша.

### Пакет сборки PlatformIO для ESP32-C6

Установленный локально PlatformIO `espressif32` 7.0.1 выбирает пакет
Arduino-ESP32, в котором отсутствует `platformio-build-esp32c6.py`. Сборка
останавливается ещё до компиляции исходников проекта. `platformio.ini` сохранён
для последующего ремонта окружения, а проверенная сборка выполняется локальным
Arduino CLI с Arduino-ESP32 3.3.10.

### Первая компиляция через Arduino CLI

Первая компиляция обнаружила три проблемы переносимости:

- Методы доступа TinyGPSPlus 1.0.3 не являются `const`, поэтому метод расчёта эпохи не мог быть `const`.
- Упакованный пакет статуса имел размер 30 байт вместо заданных 32.
- В Arduino-ESP32 3.3.10 у `fs::File` нет метода `truncate()`.

Первые две ошибки исправлены напрямую. Восстановление хвоста теперь копирует
валидный префикс во временный файл и переименовывает его, не используя
непереносимый API файловой системы.

### Несовместимые версии TypeScript и vue-tsc

Первая установка зависимостей 15 июля 2026 года выбрала TypeScript 7.0.2.
`vue-tsc` 3.3.7 обращался к неэкспортируемому пути `typescript/lib/tsc`, поэтому
проверка типов не запускалась. TypeScript зафиксирован на версии 5.9.3; в
дальнейшем его версию следует обновлять только вместе с проверкой `vue-tsc`.

### Отсутствовавшие тесты веб-протокола

Первый запуск Vitest завершился ошибкой «No test files found». Добавлены тесты
семибайтового пакета настроек, 32-байтового статуса и 28-байтовой GPS-точки.

### Типы окружения Vite

После фиксации TypeScript проверка выявила отсутствующие типы Node.js и
использование `Symbol.asyncDispose` в актуальном Vite. В конфигурацию добавлены
типы Node.js и библиотека `ESNext`. Кроме того, отправляемый Web Bluetooth буфер
явно копируется в обычный `ArrayBuffer`, чтобы исключить несовместимость с
`SharedArrayBuffer` в строгих типах TypeScript.

### Выравненная, но повреждённая запись SD

Первый вариант восстановления хвоста проверял только кратность размера файла 48
байтам. Полностью записанный, но повреждённый CRC последней записи оставался бы
в середине файла после следующей дозаписи. Восстановление изменено: теперь все
записи текущего файла последовательно проверяются по magic, версии, типу и CRC,
а файл обрезается копированием до первой невалидной записи.

### Кнопка работала только как источник пробуждения

Первый вариант настраивал GPIO5 только как источник выхода из Deep Sleep. Нажатие
во время таймерного цикла не включало выключенные экран и BLE. Добавлен детектор
фронта в основном цикле и независимый 30-секундный срок принудительного окна.
Заодно окно отделено от общего `awakeTimeSec`, чтобы экран не оставался включённым
до конца длинного цикла бодрствования.

### Белая страница GitHub Pages

GitHub Pages первоначально публиковал корень репозитория через Jekyll, тогда как
готовый интерфейс находился в `web/dist`. После первого исправления возникла гонка:
старый Jekyll-деплой завершился позже нового workflow и снова заменил страницу.
Pages переведён в режим `workflow`, затем production-сборка опубликована повторно.

### Web Bluetooth отличался от проверенной старой версии

Новый клиент фильтровал системный выбор устройств по пользовательскому UUID,
не восстанавливал ранее выданное разрешение и всегда писал команды без ответа.
Кроме того, при частично открытом GATT соединение считалось готовым до обнаружения
характеристик, из-за чего переподключение могло остановиться навсегда.

Клиент переведён на `acceptAllDevices + optionalServices`, добавлены
`navigator.bluetooth.getDevices()`, повторное обнаружение характеристик,
очередь GATT-записей и `writeValueWithResponse`. Поведение покрыто отдельными
тестами. Фактический системный диалог и радиообмен всё равно нужно подтвердить на
Oppo Find X7 Ultra с включённым трекером: браузерная автоматизация не может принять
системное разрешение Bluetooth вместо пользователя.

### На странице отсутствовал mobile viewport

Исходный `index.html` содержал только корневой `div` и скрипт Vite. Chrome на
Android мог использовать широкую виртуальную область страницы, поэтому CSS
мобильного breakpoint не соответствовал физическому экрану. Добавлен полный HTML
документ с `width=device-width`, `initial-scale=1` и `viewport-fit=cover`.

### Экспорт трека был скрыт за ручной синхронизацией

Преобразование C6T2 в GPX и GeoJSON существовало, но кнопки были коротко названы
`GPX` и `GeoJSON` и оставались выключенными до отдельного ручного скачивания всего
трека. Пользователь не мог понять, что экспорт поддерживается.

Кнопки переименованы в «Скачать GPX» и «Скачать GeoJSON». Теперь обработчик сам
проверяет локальные блоки, при необходимости продолжает BLE-передачу, проверяет
C6T2 и CRC, применяет телефонный фильтр и запускает скачивание. Завершённые
локальные треки доступны для экспорта без Bluetooth.

### Старые настройки нельзя переносить буквально

В предыдущей прошивке фильтрация GPS выполнялась на устройстве. Её буквальный
перенос нарушил бы принятые требования новой версии, поэтому минимум спутников и
HDOP перенесены как телефонные фильтры. Позднее по отдельному требованию возвращён
только параметр `pointsBeforeSleep`; он считает все валидные координаты, а
`awakeTimeSec` ограничивает исключительно ожидание первой точки.

### Подсветка иногда оставалась включённой во сне

Одного `ledcWrite(0)` оказалось недостаточно: PWM-периферия и состояние GPIO могли
сохраниться на границе Deep Sleep. Перед сном канал теперь отключается, вывод
принудительно устанавливается в низкий уровень и удерживается. ESP32-C6 использует
удержание отдельного GPIO в Deep Sleep, поэтому недоступные для этой цели общие
`gpio_deep_sleep_hold_*` были удалены после ошибки компиляции.

### Полноэкранный режим выдавал необработанную ошибку

Встроенный браузер отклонил `requestFullscreen()` с `Permissions check failed`.
Вызов обёрнут в обработку ошибки; при запрете автоматически включается CSS-режим
на весь viewport. Повторное нажатие корректно сворачивает карту.

### JBR Android Studio оказался неполным JDK

Найденный JBR 21 запускал Gradle, но не содержал `jlink`, необходимый Android
Gradle Plugin. Для воспроизводимой сборки использован полный Temurin JDK 21.
Android SDK подтверждён по пути `D:\\Android_SDK`.

### COM4 был временно занят перед прошивкой

Первая попытка загрузки завершилась до начала записи с `Access denied`: COM4 был
занят зависшим дескриптором. После повторной проверки порта дескриптор освободился,
повторная загрузка прошла успешно. Все записанные области подтверждены проверкой
hash, поэтому первая попытка не повредила содержимое flash.
